# Assignment 9 Reflection – CampusFix Residence Maintenance System

Designing the domain model and class diagram for CampusFix was an important step in moving the project from analysis and workflow thinking into structured object-oriented design. In the earlier assignments, the main focus was on understanding the problem, identifying stakeholders, gathering requirements, modeling use cases, planning Agile work, and showing system behavior through state and activity diagrams. Assignment 9 required me to think differently. Instead of focusing mainly on what the system does, I had to focus on what the system is made of in terms of core objects, their data, their responsibilities, and how they relate to each other.

One of the main challenges I faced was deciding the correct level of abstraction for the domain model. It was tempting to include too many entities because a real residence maintenance system can involve many actors and records, such as notifications, attachments, reports, maintenance categories, audit logs, and authentication details. However, I had to keep the model focused on the core problem that CampusFix is meant to solve. I therefore selected entities that directly support the main maintenance process: Student, Residence, Room, MaintenanceRequest, Technician, Admin, and MaintenanceUpdate. This was a trade-off between realism and clarity. If I included too many entities, the diagram would become difficult to read and might lose marks for complexity without purpose. If I included too few, the model would appear incomplete. I learned that a good domain model should capture the essential business structure without becoming cluttered.

Another challenge was defining the relationships correctly. In object-oriented design, it is not enough to simply say that classes are linked; the type of relationship matters. I had to think carefully about whether certain links should be associations, compositions, or aggregations. For example, I chose composition between Residence and Room because a room is meaningfully part of a residence. I also chose composition between MaintenanceRequest and MaintenanceUpdate because updates exist as part of the request history and do not make much sense independently. On the other hand, the relationships between Student and MaintenanceRequest, Technician and MaintenanceRequest, and Admin and MaintenanceRequest were modeled as associations because these actors interact with requests but do not own them in the same structural sense.

Method definition was another area that required careful thought. Earlier assignments already described functions like submitting requests, assigning technicians, updating progress, resolving issues, and confirming repairs. In this assignment, I had to convert those actions into methods that belong to the right classes. This forced me to think about responsibility allocation. For example, submitRequest() belongs naturally to Student, while assignTechnician() belongs to Admin, and completeRepair() belongs to Technician. MaintenanceRequest itself also needed lifecycle methods such as approve(), reject(), markResolved(), closeRequest(), and reopen(). This exercise helped me understand that class diagrams are not only about data fields but also about behavior attached to the right object.

This assignment aligned strongly with the previous CampusFix assignments. The functional requirements from Assignment 4 shaped the classes and methods because the system must support issue reporting, request review, assignment, progress tracking, and closure. The use cases from Assignment 5 guided the operations shown in the classes. The Agile user stories and sprint planning from Assignment 6 helped confirm which features are central to the minimum viable product. Assignment 8 was especially useful because the state diagram for a maintenance request directly influenced the MaintenanceRequest class. The request status field and methods such as submit(), approve(), assignTechnician(), markResolved(), closeRequest(), and reopen() came naturally from the previously modeled object lifecycle.

A key trade-off I made was keeping Admin and Technician as separate classes instead of combining them into a general Staff class. From a pure inheritance perspective, a shared parent class might reduce duplication because both have names, emails, and system roles. However, I decided that for this assignment, keeping them separate made the diagram clearer and easier to explain. Since the lecturer is marking clarity, traceability, and correctness, separate classes show the workflow roles more directly. This was a practical design choice rather than a purely theoretical one.

The biggest lesson I learned from this assignment is that object-oriented design requires balance. A class diagram should not be a random technical sketch; it should reflect the business domain accurately, support the required system behavior, and remain understandable. I also learned that earlier analysis artifacts are not isolated tasks. Requirements, use cases, user stories, and behavioral diagrams all feed into structural design. Overall, Assignment 9 helped me see how software engineering artifacts connect together into one coherent system design. It made the CampusFix project feel more like a real software product rather than just a collection of separate assignment documents.
