# Assignment 8 Reflection  
## CampusFix Residence Maintenance System

## Overview
This reflection discusses the challenges experienced while developing state transition diagrams and activity diagrams, as well as the insights gained during the modeling of system behavior.

---

## Challenges Experienced

### 1. Choosing the Right Level of Detail
One of the primary challenges was determining the appropriate level of detail for both state diagrams and activity workflows.

Initially, there was a tendency to:
- Either oversimplify the system behavior, or  
- Include too many states and transitions, making diagrams complex and difficult to read  

Balancing clarity and completeness required careful consideration of what constitutes a meaningful state versus a minor system action.

---

### 2. Differentiating Between State Diagrams and Activity Diagrams
Another challenge was clearly distinguishing between:

- **State Diagrams (Object Behavior)**  
- **Activity Diagrams (Process Flow)**  

State diagrams focus on how a single object changes over time, while activity diagrams represent workflows involving multiple actors.

At first, there was overlap between the two, but refining the understanding helped ensure:
- State diagrams model lifecycle transitions  
- Activity diagrams model step-by-step processes  

---

### 3. Aligning Diagrams with Previous Assignments
Ensuring consistency with:
- Functional Requirements (Assignment 4)  
- Use Cases (Assignment 5)  
- User Stories (Assignment 6)  

was challenging but essential.

Each diagram needed to:
- Reflect actual system requirements  
- Avoid introducing new, undefined behaviors  

This required revisiting earlier work and ensuring traceability across all artifacts.

---

### 4. Managing Workflow Complexity
Some workflows, such as technician assignment and issue resolution, involve:
- Multiple decision points  
- Multiple actors  
- Possible alternate paths  

Designing these workflows in a way that remained readable while still accurate required:
- Breaking processes into logical steps  
- Using clear decision points  
- Avoiding unnecessary duplication  

---

## Key Learnings

### 1. Importance of Modeling Before Implementation
This assignment reinforced the importance of fully understanding system behavior before development begins.

State and activity diagrams help:
- Identify missing logic  
- Clarify system interactions  
- Improve overall system design  

---

### 2. Improved Understanding of System Dynamics
The modeling process provided deeper insight into:
- How requests flow through the system  
- How users interact with the system  
- How different components depend on each other  

---

### 3. Value of Traceability
Linking diagrams to requirements and user stories demonstrated the importance of traceability in software engineering.

It ensures:
- Every feature has a purpose  
- No functionality is implemented without justification  
- Stakeholder needs are consistently addressed  

---

### 4. Difference Between Static and Dynamic Modeling
This assignment highlighted the difference between:

- **Static Models** (Use Cases, Requirements, Architecture)  
- **Dynamic Models** (State and Activity Diagrams)  

Dynamic modeling provides a clearer view of:
- System behavior over time  
- Process execution  
- Real-world interaction flow  

---

## Conclusion

Overall, this assignment enhanced understanding of system behavior modeling and reinforced the importance of aligning system design with requirements and Agile planning.

The experience gained will be valuable in future system design and implementation, particularly in ensuring that systems are both logically sound and aligned with stakeholder expectations.
